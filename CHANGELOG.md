# Changelog

## 3.0.2

- Added `git_untrack` tool: removes files from the git index without deleting them from disk (`git rm --cached`). Accepts a `paths` array or `path` string shorthand. Use this to stop tracking files that should be `.gitignored`. Wired into task-profile detection (`LooksLikeUntrackTask`) with keywords such as `untrack`, `git rm`, `rm --cached`, `stop tracking`, `gitignore`, and `exclude from git`; appears in `recommend_tools` results and `DefaultRecommendedGitToolNames`. Also mentioned in the bound-session hint so models know to prefer it over shell `git rm --cached`.
- Fixed bridge silently re-binding to a different VS instance when the previously bound instance closes. `BridgeConnection.RetryAfterFailureAsync` now calls `VsDiscovery.IsProcessGone` after evicting the cached instance on a pipe failure. If the VS process has exited, a `SessionLostError` (-32005) is thrown with the label and PID of the closed instance and instructions to call `bridge_health` then `bind_solution` or `bind_instance` to reconnect — instead of silently auto-discovering and binding to whatever VS instance happens to be open next. Added `SessionLostError = -32005` to `BridgeConnectionDefaults` and `IsProcessGone(int processId)` helper to `VsDiscovery`.
- Fixed `set_version` tool silently claiming success when a version regex did not match a target file. `UpdateFile` in `SetVersionTool.cs` was unconditionally adding every existing file to the `updated_files` result, even when the regex found nothing to replace and no write occurred. The method now takes a `hasMatch` predicate; files whose pattern is not found are collected into a `skipped_files` list and surfaced as a `warning` in the tool output rather than being reported as updated.

## 3.0.1

- Added `cmake_configure` tool: runs cmake to configure a project from a source directory. Accepts `preset` (uses `--preset`) or `build_dir` (uses `-S`/`-B`), plus optional `source_dir` (defaults to the bound solution's repository root), `extra_args` string array, and `timeout_seconds` (default 180). Resolves `cmake.exe` from PATH, VS-bundled locations (VS 2026, 2022, and 2019 across Enterprise/Professional/Community/BuildTools), and `C:\Program Files\CMake\bin`. Implemented via the new `CmakeRunner` helper (modelled after `GhRunner`) and registered in the `project` tool category.
- Added `cmake_cache` tool: reads the CMake cache for a given `build_dir` and returns all cache variables. Accepts an optional `filter` substring to narrow results. Useful for inspecting configured paths, flags, and options without opening `CMakeCache.txt` directly.
- Fixed BP1012 best-practice warning in `DiagnosticsTools.cs` (file exceeded 1000-line threshold) by extracting `CreateCompileFileTool`, `EnrichCompileFileResponse`, `IsHeaderCompileTarget`, `CreateRunCodeAnalysisTool`, and `TryCaptureAnalysisDiagnosticsAsync` into a new `DiagnosticsCompileTools.cs` partial class. Both files now stay comfortably under the threshold.
- Fixed `apply_diff` `edits` array silently counting duplicate/already-applied edits as failures. `IsPatchContentMismatch` in `SafetyHelpers.cs` now catches `"old_content was not found"` errors (produced by search-block / `file`+`old_content`+`new_content` patches) in addition to the existing `"mismatch"` check (unified diff hunks), enabling the `alreadySatisfied` detection path to fire for the common targeted-edit format.
- Fixed `apply_diff` `edits` array not propagating `alreadySatisfied` in per-entry results. `ApplyMultipleEditsAsync` in `DocumentTools.cs` now reads `Data.items[0].alreadySatisfied` from the underlying patch response and includes `"alreadySatisfied": true` in the per-edit entry, so callers can distinguish freshly-applied edits from those already in the desired state.
- Added `path` string shorthand parameter to `git_add`, `git_restore`, `git_reset`, `git_diff_unstaged`, and `git_diff_staged`. Previously all five tools only accepted a `paths` array; callers can now pass `path: "src/Foo.cs"` as a convenience alias for `paths: ["src/Foo.cs"]`. `git_add` and `git_restore` also relaxed their `paths` parameter from required to optional so either form is accepted independently. New `GetEffectivePathList` and `GetEffectivePathListOrNull` helpers in `GitTools.cs` implement the coalescing logic (`paths` takes precedence; `path` is the fallback; absent → `["."]` or `null`).
- Fixed BP1025 (`warnings`/`errors` best-practice analyzer) emitting false positives for C/C++ local variable declarations and identifiers inside line comments. `IsInsideCppLineComment` now skips any regex match whose position is preceded by `//` on the same line; `IsInsideFunctionParams` walks backwards from the match and skips it unless an unmatched `(` is found before a statement boundary (`; { }`), eliminating false positives like `std::string code, comment;` (local declaration) and `vector a,b` inside comment text.
- Fixed `warnings`, `errors`, and `messages` `path` filter silently matching nothing on Windows when the caller used forward slashes (e.g. `path: "src/libslic3r"`). `DiagnosticRow.MatchesPath` now delegates to `ContainsPath`, which normalises both sides to `/` before comparing, so forward- and back-slash path filters both work against Windows-style stored paths.
- Fixed `warnings`, `errors`, and `messages` passive-cache responses embedding the full unfiltered diagnostic snapshot (all severity buckets with every row) in `Cache.snapshot`, causing responses to exceed 2 MB when thousands of warnings were present. `DocumentDiagnosticsSnapshot.ToJson` now calls `DiagnosticBucket.ToCountSummaryJson` instead of `ToDataJson`, emitting only the pre-existing count/metadata fields (no rows). The full rows are already present in the main `Data` object.

## 3.0.0

- Added `run_tests` tool: runs `dotnet test` from the bound solution directory and returns structured pass/fail results. Supports project selection (by name or path, defaulting to the active solution), configuration, framework, runtime, VSTest `--filter` expressions, loggers (`trx`, etc.), results directories, data collectors, verbosity, `--blame`, and a configurable timeout (default 120 s). The response includes a `testSummary` object with `outcome`, `passed`, `failed`, `skipped`, and `total` counts parsed from the `dotnet test` summary line, plus truncated `stdout`/`stderr` controlled by `head_lines`, `tail_lines`, and `max_lines`. `no_restore` defaults to `true`. Aliased as `test`, `dotnet_test`, `unit_tests`, `run_unit_tests`, and `vstest`.
- Fixed `apply_diff` for non-project files (`<MiscFiles>` like `CHANGELOG.md`, `.ps1`, `.txt`): `read_file` now registers and returns an `f:N` handle in `Data.handle` so models can pass it directly to `apply_diff` without a separate `find_files` call. `apply_diff` also now accepts plain paths (absolute or solution-relative) in addition to handles, so local models without the host `Edit` tool can edit any file with just `read_file` + `apply_diff`. Removed the handle-only restriction from both the single-edit and multi-edit (`edits` array) code paths. Updated tool descriptions and MCP initialize instructions accordingly.

- Fixed `github_pr_list`, `github_pr_view`, and `github_pr_diff` hanging for exactly 30 seconds when spawned from a service or background host. `GhRunner.CreateStartInfo` now sets `RedirectStandardInput = true` and closes stdin immediately after process launch so `gh` receives EOF and knows it is non-interactive; previously `gh` inherited the parent's stdin and blocked indefinitely waiting for interactive input (browser auth prompt, update check, etc.). Added `GH_PROMPT_DISABLED=1` and `BROWSER=""` environment variables as belt-and-suspenders to suppress any remaining interactive behaviour.
- Fixed `errors`, `warnings`, and `messages` reporting empty `code`, `path`, `project`, `file`, and `text` fields in the `filter` echo after service-side content filtering. The VSIX only receives severity/quick/max; content filters are applied service-side by `DiagnosticCollection`. The new `PatchFilterEcho` method in `DiagnosticsTools` overwrites the stale filter fields in the VSIX response with the actual `DiagnosticQueryOptions` that were applied, so the echo matches what was filtered.
- Added `path` parameter to `git_log` to limit history to commits that touched a specific file or directory, using LibGit2Sharp's `repo.Commits.QueryBy(path)` file-history walk (equivalent to `git log -- <path>`). Added `paths` array parameter to `git_diff_unstaged` (case-insensitive substring filter on changed paths via LibGit2Sharp) and `git_diff_staged` (appends `-- <paths>` to the git diff arguments).
- Fixed `git_push` and `git_pull` producing double-space arguments when optional remote or branch parameters are omitted. Replaced `$"push {a} {b} {c}".TrimEnd().Replace("  ", " ")` (single-pass, only collapses one pair of spaces) with `string.Join(" ", parts.Where(s => !string.IsNullOrEmpty(s)))`. Fixed `git_stash_list` being registered under `GitDiffAndMetaTools` instead of `GitStashTools`. Removed the unused `BuildPathList` dead-code helper.
- Merged PR #11 from contributor Henry Li (henrli): VS 2026 compatibility fixes and new `list_launch_profiles` / `set_launch_profile` commands (see following entries).
- Added VS 2026 compatibility: updated VSSDK Build Tools to 18.5 for VS 2026 VSIX packaging, marked `Newtonsoft.Json` and `LanguageServices` as compile-only to avoid binding conflicts with VS-provided assemblies, excluded Roslyn, Threading, Composition, and other VS-shipped DLLs from VSIX output, fixed `VisualStudioMajorVersion` (`"18"` → `"2026"` to match the folder name change), and added `global.json` for SDK build tools resolution.

## 2.2.12

- Made `http_enable` and `http_disable` reconcile the service-owned HTTP listener instead of only toggling local process state. When called from a short-lived stdio MCP child the toggle now reaches the long-running service over the control pipe, so the listener on `localhost:8080` actually starts or stops and the persisted enable flag stays in sync. Falls back to the in-process listener when the service is not installed.
- Fixed the service control pipe accepting only one client at a time. A single long-lived control client (such as the stdio MCP child sending `MCP_REQUEST`) was blocking every other process — including `http_enable`/`http_disable` — from connecting. The accept loop now hands each connection to a per-client task and immediately loops back to accept the next.
- Made `http_status` cross-process honest. It now reports the live `localhost:8080` listener probe alongside the persisted enable flag, and surfaces a `note` describing what reconciliation actually did (service confirmed listener up / down, or fell back to in-process).
- Added `HttpServerController.IsPortListening` as the cross-process truth source so callers can tell whether a listener is really accepting connections versus only consulting the in-process `IsRunning` flag (which is process-local and now documented as such).
- Hardened `HttpServerController.Disable()` to handle `IOException` and `UnauthorizedAccessException` when deleting the flag file instead of letting them bubble up and abort the disable path.
- Centralized the service control pipe name in `NamedPipeAccessDefaults.ServiceControlPipeName` so the VSIX, the service, and the control client share a single source of truth, and added `NotifyHttpEnableAsync` / `NotifyHttpDisableAsync` to `ServiceControlClient` for callers that prefer the typed wrapper.
- Refreshed the `http_enable`, `http_disable`, and `http_status` tool descriptions in the catalog to describe the service-reconciliation behavior.
